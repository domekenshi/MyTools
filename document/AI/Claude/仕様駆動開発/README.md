# Claude Code で仕様駆動開発をやる

Kiro のような「コードの前に仕様を書く」ワークフローを、Claude Code で再現するための手順書。
Kiro は専用IDEで `requirements.md → design.md → tasks.md` を強制するが、Claude Code では
**同じ3点セットを自分で運用する**ことで近いことができる。

## 全体の流れ

```
1. requirements.md  要件を固める（何を作るか・なぜ）
        ↓ 承認
2. design.md        設計を固める（どう作るか）
        ↓ 承認
3. tasks.md         実装タスクに分解（手順）
        ↓ 承認
4. 実装             tasks を1つずつ消化
```

各ステップで Claude に「次に進む前に内容を確認させて」と伝え、承認してから次へ進むのがコツ。
これが Kiro の「レールから外れない」感覚を手動で作る部分。

## Claude Code 側の道具との対応

| Kiro の概念 | Claude Code での実現 |
|---|---|
| steering files | CLAUDE.md（プロジェクト指示） |
| spec（req/design/tasks） | このフォルダの3ファイルを手動運用 |
| spec を書いてから実装 | Plan モードで計画→承認→実装 |
| tasks の進捗管理 | TodoWrite（タスクの消化を可視化） |
| hooks（保存時lint/test） | settings.json の hooks |

## 使い方（プロンプト例）

### ステップ1: 要件
```
○○機能を作りたい。まず requirements.md を書いて。
ユーザーストーリーと受け入れ条件（Acceptance Criteria）を含めて。
コードはまだ書かないで、要件だけ。
```

### ステップ2: 設計
```
requirements.md を承認します。次に design.md を書いて。
アーキテクチャ・データモデル・主要な処理フロー・代替案の検討を含めて。
```

### ステップ3: タスク分解
```
design.md を承認します。次に tasks.md を書いて。
実装を小さいタスクに分解して、依存順に並べて。各タスクはチェックボックスで。
```

### ステップ4: 実装
```
tasks.md の上から順に実装して。1タスクごとに止まって確認させて。
```

## テンプレート

このフォルダに各ステップのテンプレートを置いてある：

- [templates/requirements.md](templates/requirements.md)
- [templates/design.md](templates/design.md)
- [templates/tasks.md](templates/tasks.md)

新しい機能を作るときは、これをコピーして埋めていく。

## 他プロジェクトで使い回す（ここが本題）

新しいプロジェクトを始めるたびに Claude に説明し直さなくて済むように、置き場所を決めておく。
方法は2つ。**A をやっておけば全プロジェクトで自動的に効く**ので基本は A 推奨。

### 方法A: グローバル設定で全プロジェクトに自動適用（おすすめ）

`~/.claude/CLAUDE.md`（ホームの Claude 設定）は **すべてのプロジェクトで読み込まれる**。
ここに方針を書いておけば、どの新規プロジェクトでも Claude が最初からこの流れを提案する。

`~/.claude/CLAUDE.md` に以下を追記：

```markdown
## 開発の進め方（全プロジェクト共通）

機能を実装するときは、いきなりコードを書かず
requirements → design → tasks の順で仕様を固めてから実装する。
各ステップは内容を確認・承認してから次へ進む。
テンプレートは ~/.claude/spec-templates/ にある。
```

テンプレート本体も共通の場所に置いておく（1回だけ）：

```bash
mkdir -p ~/.claude/spec-templates
cp templates/*.md ~/.claude/spec-templates/
```

これで新規プロジェクトでは **何もコピペせずに**「○○機能を作りたい」と言うだけで
Claude が requirements から始めてくれる。

### 方法B: プロジェクトごとに置く（チーム共有したいとき）

リポジトリにテンプレートと方針を含めて、チーム全員で同じ流れにしたい場合。
新規プロジェクトのルートで：

```bash
# テンプレートを置く
mkdir -p docs/spec
cp ~/.claude/spec-templates/*.md docs/spec/

# プロジェクトの CLAUDE.md に方針を追記（下のスニペットをコピペ）
```

プロジェクト `CLAUDE.md` に貼るスニペット：

```markdown
## 開発の進め方

機能を実装するときは requirements → design → tasks の順で仕様を固めてから実装する。
テンプレートは docs/spec/ を参照。各ステップは承認してから次へ進む。
```

### どちらを使う？

| | 方法A（グローバル） | 方法B（プロジェクト） |
|---|---|---|
| 自分専用・全プロジェクトで楽したい | ◎ | △ |
| チームで流れを揃えたい | ✕（自分にしか効かない） | ◎ |
| コピペ作業 | 不要（初回のみ） | 毎回少し必要 |

→ **個人なら A、チーム開発なら B**（A と B は併用してもよい）。

### 特定フォルダだけ仕様駆動を外す（上書き）

方法A はすべてのプロジェクトに効くので、アイデア置き場や使い捨ての実験フォルダでは
「毎回 requirements から」が重く感じることがある。
その場合は **そのフォルダの `CLAUDE.md` に上書き指示を書く**。
より近い（深い）場所の CLAUDE.md が優先されるので、グローバル設定を打ち消せる。

例（実験フォルダの `CLAUDE.md` に追記）：

```markdown
## 開発の進め方

ここは軽い実験・アイデア置き場なので、グローバル設定の
仕様駆動（requirements → design → tasks）は任意とする。サッと試したいときは省略してよい。
個別プロジェクトに育てる段階で改めて仕様駆動を適用する。
```

ポイント: グローバル = 既定の動き、プロジェクト = その場の例外、という関係。
「全体は仕様駆動 ON、ここだけ OFF」のように使い分けられる。

## Kiro と比べた割り切り

- ✕ IDEによる強制力はない（自分で「仕様→実装」の順を守る必要がある）
- ◯ どのIDE・CLIでもできる / フォーマットを自由にカスタムできる
- ◯ 慣れたら CLAUDE.md に「機能追加時は必ず requirements→design→tasks の順で」と
  書いておけば、Claude が自発的にこの流れを提案するようになる
